## Overview

CLOPR2 uses **pgBackRest** with **Azure Blob Storage** as the backup and PITR infrastructure for PostgreSQL 16. This is additive to the existing streaming replication (on-prem HA + Azure DR) and does not replace it.

```
pg-primary (10.0.96.11)
    │
    ├── Streaming replication ──→ pg-standby (10.0.96.14)   [on-prem HA]
    ├── Streaming replication ──→ vm-pg-dr-fce (10.200.0.2)  [Azure DR]
    │
    └── pgBackRest WAL archive ──→ Azure Blob Storage
            account: clopr2backupkatar
            container: pgbackrest
            path: /pgbackrest
```

**Why pgBackRest:**
- Industry-standard PostgreSQL backup tool (maintained by PgOSE / CrunchyData)
- Native Azure Blob Storage support (no intermediary tooling)
- WAL archiving enables arbitrary point-in-time recovery
- Backup compression (3.9MB for 29.4MB database) and integrity checksums
- Retention management built-in

### pgBackRest Configuration

File: `/etc/pgbackrest/pgbackrest.conf` on pg-primary
Permissions: `640`, owned `root:postgres`

```ini
[global]
repo1-type=azure
repo1-path=/pgbackrest
repo1-azure-account=clopr2backupkatar
repo1-azure-key=<storage-account-key>
repo1-azure-container=pgbackrest
repo1-retention-full=2
log-level-console=info
log-level-file=detail
log-path=/var/log/pgbackrest
process-max=1

[main]
pg1-path=/var/lib/postgresql/16/main
```

Stanza name: `main` (matches PostgreSQL cluster name)

### WAL Archiving

Configured in `/etc/postgresql/16/main/postgresql.conf`:

```
archive_mode = on
archive_command = 'pgbackrest --stanza=main archive-push %p'
wal_level = replica   # pre-existing; sufficient for archiving
```

A PostgreSQL restart is required when `archive_mode` changes. After the restart, every WAL segment generated by pg-primary is automatically pushed to Azure Blob Storage via the archive_command.

## Backup Operations

### Run a Full Backup

```bash
sudo -u postgres pgbackrest --stanza=main --type=full backup
```

### Run a Differential Backup (after at least one full)

```bash
sudo -u postgres pgbackrest --stanza=main --type=diff backup
```

### View Backup Inventory

```bash
sudo -u postgres pgbackrest --stanza=main info
```

### Verify Archive Pipeline

```bash
sudo -u postgres pgbackrest --stanza=main check
```

This pushes a test WAL segment and verifies it lands in Azure successfully.

## PITR — Point-In-Time Recovery

### How It Works

1. pgBackRest takes a base backup (full copy of the data directory)
2. Every WAL segment generated after the backup is archived to Azure
3. To recover to time T:
   - Restore the base backup taken before T
   - Apply WAL segments from the archive until T is reached
   - PostgreSQL enters recovery mode and replays transactions

### Recovery Time Estimate

| Step | Time |
|------|------|
| Download backup from Azure (29.4MB → local) | ~90 seconds |
| WAL replay (depends on time gap) | Seconds to minutes |
| Total | <5 minutes for recent recovery |

### Recovery Procedure

**Prerequisites:** pg-primary is stopped or unavailable; restore target is a clean host.

```bash
# 1. Stop PostgreSQL (if still running)
sudo systemctl stop postgresql

# 2. Clear the data directory
sudo rm -rf /var/lib/postgresql/16/main/*

# 3. Restore the latest backup
sudo -u postgres pgbackrest --stanza=main restore

# 4. For PITR to a specific timestamp:
sudo -u postgres pgbackrest --stanza=main restore \
  --target="2026-03-18 09:00:00" \
  --target-action=promote

# 5. Start PostgreSQL — it will enter recovery mode and replay WAL
sudo systemctl start postgresql

# 6. Monitor recovery
sudo -u postgres psql -c "SELECT pg_is_in_recovery(), now();"
```
