01 · Executive Summary

What was built and validated

Project

A production-grade hybrid disaster recovery platform spanning Proxmox on-premises infrastructure and Microsoft Azure. Built entirely with Infrastructure-as-Code (Terraform + Ansible), secured with WireGuard VPN, and validated end-to-end with operational evidence.

What was validated

  • ✔ On-prem HA: VIP failover/fallback via Keepalived VRRP — RTO <1s
  • ✔ Full-site failover from on-prem to Azure DR VM — RPO 0 bytes
  • ✔ Full-site failback from Azure back to on-prem — RTO 20m 53s
  • ✔ Application health endpoint verified at every stage
  • ✔ 33 evidence files committed to git on main

IaC stack

Terraform (azurerm + bpg/proxmox) for all cloud and on-prem VM provisioning. Ansible for PostgreSQL HA, Keepalived, WireGuard, and app deployment.

Observability

Azure Monitor + Log Analytics workspace. Azure Arc agents on all 3 on-prem VMs (Connected). Application health endpoint at /health used as primary DR acceptance gate.

Security

WireGuard VPN tunnel (UDP 51820), NSG least-privilege, Azure Key Vault for secrets, Defender for Cloud, RBAC, and network threat model documented.

PostgreSQL HA

Streaming replication: pg-primary → pg-standby (on-prem HA) + pg-primary → Azure DR VM. Keepalived VRRP moves VIP in <1s when primary stops advertising.

WireGuard tunnel

pg-primary (10.200.0.1) ↔ vm-pg-dr-fce (10.200.0.2). Persistent keepalive 25s. Azure public IP: 20.216.128.32 UDP 51820. Tunnel carries replication traffic.

Keepalived VRRP

VIP 10.0.96.10, VRID 51, nopreempt. Failover trigger: systemctl stop keepalived. Fallback: automatic on restart (priority 100 > 90).

Failover RTO note — SSH interruption

The operational failover RTO of 48m 42s was caused by WSL losing its network route to PVE mid-execution after the SSH ControlMaster socket was cleared. The remaining steps were completed via Azure run-command (significant per-call latency). The promotion itself succeeded; only the confirmation step was delayed. Under uninterrupted SSH access, the estimated clean RTO is <5 minutes. All acceptance criteria were met.

07 · Follow-up Items

Post-drill maintenance

🔄
Rebuild pg-standby (10.0.96.14) as streaming replica
pg-standby is active but stuck at LSN 0/542FD18 — the pre-failover timeline. When pg-primary was stopped during full-site failover, pg-standby lost its replication feed and cannot automatically rejoin the new timeline after pg-primary was promoted from the DR VM. Fix: run pg_basebackup on pg-standby from the restored pg-primary, then start PostgreSQL with standby.signal. Not a gate item — on-prem HA degrades to single-node primary until resolved.
📋
Document SSH ControlMaster socket management in runbook
The WSL SSH ControlMaster socket at ~/.ssh/ctl/pve goes stale when WSL changes network. Add a mandatory pre-execution check: rm -f ~/.ssh/ctl/pve && ssh pve echo ok to both the failover and failback runbooks as a pre-step.
💰
Azure auto-shutdown re-enabled on vm-pg-dr-fce
Auto-shutdown was disabled during the DR drill to prevent the DR VM from being stopped mid-execution. Verify it has been re-enabled at the desired schedule to control Azure costs.
